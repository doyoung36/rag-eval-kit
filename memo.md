# RAGAS 평가 지표 정리

RAG 파이프라인을 **검색(retrieval)** 과 **생성(generation)** 두 단계로 나눠 평가한다.
RAGAS의 대표 네 지표가 각 단계의 품질을 짝지어 측정한다.

| 지표 | 비교 대상 | 평가 단계 | 채점 방식 | reference 사용 |
|------|----------|----------|----------|:---:|
| Faithfulness | 답변 ↔ 검색 문맥 | 생성 | LLM (주장 분해 → 대조) | ✗ |
| Answer Relevancy | 답변 ↔ 질문 | 생성 | LLM(역질문) + 임베딩 | ✗ |
| Context Precision | 검색 청크 ↔ 정답 (순위 고려) | 검색 | LLM | ✓ |
| Context Recall | 정답 ↔ 검색 청크 | 검색 | LLM | ✓ |

> 네 지표 중 셋(Faithfulness, Answer Relevancy, Context Recall)과 Context Precision까지
> 사실상 대부분이 **LLM이 채점**한다. 점수는 결정론적 계산이 아니라 **판정 LLM의 품질에
> 좌우**되며, 로컬 소형 모델이나 비영어 데이터에서는 변동이 크다.

---

## 1. Faithfulness (충실도) — 답변이 문맥에 근거하는가

답변이 검색된 문맥에 사실적으로 근거하는지(환각 여부)를 본다.

1. LLM이 **답변을 검증 가능한 주장(statement) 단위로 분해**한다. 이때 질문도 단서로
   함께 넣어 대명사를 풀고 문장을 완성한다.
2. 각 주장을 문맥과 대조해 *"이 주장이 문맥에서 직접 추론되는가?"* → 예(1)/아니오(0) 판정.
3. **점수 = 근거 있는 주장 수 ÷ 전체 주장 수.**

- 정답(reference)은 사용하지 않는다. 오직 답변과 검색 문맥만 비교한다.
- 낮으면 = 문맥에 없는 내용을 지어냄(환각).
- 주장이 0개 생성되면 점수는 0이 아니라 `NaN`이다. 따라서 점수가 **정확히 0**이라는 것은
  "주장은 생성됐으나 전부 문맥으로 뒷받침되지 않았다"는 의미다.

---

## 2. Answer Relevancy (답변 적절성) — 답변이 질문에 답하는가

답변이 질문에 곧장 답하는지(딴소리·장황함·회피 여부)를 본다.

1. LLM에게 **답변만 주고 "이 답변이 나오려면 어떤 질문이었을까?"** 라며 가짜 질문을
   여러 개 역생성한다.
2. 역질문들을 **임베딩**해 **원래 질문 임베딩과의 코사인 유사도**를 잰다.
3. **점수 = 유사도들의 평균.**

- 정답·문맥 모두 사용하지 않는다. 측정하는 것은 **답변 → 질문 정렬도**다.
- 낮으면 = 답변이 질문과 어긋나거나, 장황하거나, 회피적("모르겠다")임.
- 임베딩 모델이 데이터 언어(예: 한국어)와 맞지 않으면 점수가 구조적으로 낮아진다.

---

## 3. Context Precision (문맥 정밀도) — 쓸모있는 청크가 위에 왔는가

검색한 청크들 중 정답에 기여하는 것이 상위에 랭크됐는지를 본다.

1. 검색된 top-k 청크 각각에 대해 LLM이 *"이 청크가 정답을 만드는 데 관련 있는가?"* 판정.
2. 단순 비율이 아니라 **순위에 가중**한다 — 관련 청크가 상위에 있을수록 점수가 높다
   (precision@k의 평균).
3. **점수 = 순위 가중 평균.**

- 핵심은 *"노이즈는 아래로, 정답 근거는 위로"* 잘 정렬했는가.
- 낮으면 = 리트리버가 노이즈 청크를 상위에 많이 가져옴.

---

## 4. Context Recall (문맥 재현율) — 필요한 정보를 빠짐없이 검색했는가

정답을 만드는 데 필요한 정보를 검색 단계에서 누락 없이 가져왔는지를 본다.

1. **정답(reference)을 문장 단위로 분해**한다.
2. 각 문장이 검색된 문맥 안에 들어있는지 판정한다.
3. **점수 = 문맥이 커버하는 정답 문장 수 ÷ 전체 정답 문장 수.**

- 검색이 **놓친 게 있는지**를 측정한다.
- 색인(index)이 불완전하면 정답 문서가 색인에 없어 구조적으로 낮아진다.
  제대로 된 baseline은 전체 코퍼스를 색인한 뒤 재측정해야 한다.

---

## 묶어서 보는 법

- **검색 품질** = Context Precision(정렬) + Context Recall(누락)
- **생성 품질** = Faithfulness(환각) + Answer Relevancy(적절성)

검색이 좋은데(Precision·Recall 높음) 생성 지표만 낮다면, 생성 단계 문제일 수도 있지만
**지표 자체의 특성 문제**일 가능성을 먼저 의심해야 한다(아래 참고).

---

## 단답형 정답에서의 함정 (중요)

Faithfulness와 Answer Relevancy는 **단답형(고유명사·숫자·날짜) 정답에서 구조적으로 오작동**한다.
두 지표 모두 답변을 문장으로 재구성하거나 역질문하는 단계를 거치기 때문이다.

- **Answer Relevancy**: 단어 하나짜리 답변은 역질문 복원이 불안정해 점수가 발산한다.
  답변과 정답이 완전히 일치해도 0.3대가 나올 수 있다.
- **Faithfulness**: 단답을 질문과 합쳐 평서문으로 재구성하는 과정에서, 문맥이 실제로
  말한 것보다 강한 주장이 만들어져 NLI 판정에서 탈락할 수 있다.
  (예: 문맥은 "계획을 세웠다"인데 재구성된 주장은 "실행했다"가 되어 불일치)

반면 **Context Precision / Context Recall은 단답형에서도 신뢰할 만하다** — 답변 형식이 아니라
검색 결과와 정답을 비교하기 때문이다.

### 함의
- 단답형 골든셋이라면 Answer Relevancy / Faithfulness 점수를 액면 그대로 신뢰하지 말 것.
- 답변의 정오(正誤)를 보려면 **`answer_correctness`** 또는 **`SemanticSimilarity`**(answer_similarity)를
  쓰는 편이 낫다. 이들은 답변(response)과 정답(reference)을 직접 비교한다.
- Faithfulness / Answer Relevancy를 의미 있게 쓰려면 답변을 **완결된 문장 형태로 생성**하도록
  프롬프트를 설계해야 한다.

## 참고: 디버깅 팁

LLM 기반 지표가 이상한 점수를 줄 때는 **중간 산출물**을 확인하면 원인이 추정에서 확정으로 바뀐다.
Faithfulness의 경우 ① 생성된 주장 리스트와 ② 각 주장의 verdict·reason을 찍어보면
점수가 그렇게 나온 이유를 직접 볼 수 있다.
