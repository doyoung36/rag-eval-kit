**실험 목표**

RAGAS로 임베딩 모델 × 벡터DB 조합 비교 분석 (한국어 위주 corpus)

---

**확정된 스택**

| 항목      | 선택                                                 |
| --------- | ---------------------------------------------------- |
| 평가 LLM  | Claude Haiku (기본) → 애매하면 Sonnet으로 업그레이드 |
| 실험 환경 | Apple Silicon M5, 통합메모리 24GB, 셀프호스팅        |

**임베딩 모델 후보**

| 모델                 | 특징                                               |
| -------------------- | -------------------------------------------------- |
| Qwen3-Embedding-0.6B | 메인. 최신 오픈소스 최강, Apache 2.0               |
| dragonkue/BGE-m3-ko  | 한국어 특화 파인튜닝, sparse 벡터 지원, Apache 2.0 |
| embeddingGemma       | 베이스라인. 라이선스 상업용 주의                   |

→ Qwen3-VL은 멀티모달 전용이라 이번 실험 범위 밖

**벡터DB 후보**

| DB       | 포지션                                                                            |
| -------- | --------------------------------------------------------------------------------- |
| Qdrant   | 순수 벡터DB 성능 기준점. sparse 인덱스 지원 → BGE-m3-ko와 궁합 좋음               |
| pgvector | PostgreSQL 익스텐션. 운영 오버헤드 최소, ANN 성능은 가장 낮음                     |
| Neo4j    | 원래 그래프DB + 벡터 기능 추가. 조만간 그래프DB도 쓸 일 있어서 Weaviate 대신 채택 |

→ Weaviate는 복잡도 대비 실무 활용 가능성이 낮아 Neo4j로 대체

---

**실험 구조**

```
고정: 청킹 전략, 평가 데이터셋 (한국어 QA)
변수: 임베딩 모델 3개 × 벡터DB 3개 = 9조합

RAGAS 지표:
  - context_precision
  - context_recall
  - answer_relevancy
  - faithfulness
```

---

**주요 인사이트 (대화 중 나온 것들)**

- 임베딩 모델은 문서→벡터 변환(인덱싱)과 질문→벡터 변환(검색) 두 군데 모두 쓰임. 인덱싱/검색에 반드시 같은 모델 써야 함
- 벡터DB 간 성능 차이는 ANN 알고리즘(HNSW 등), 메타데이터 필터링 효율, sparse 인덱스 지원 여부에서 발생
- BGE-M3의 sparse 벡터는 Qdrant에서만 제대로 활용 가능 → 같은 모델인데 DB에 따라 RAGAS 점수가 달라질 수 있는 포인트
- sparse 기능 자체는 혁신적이었으나 생태계 지원 부족 + 최신 dense 모델 품질 향상으로 실무 활용은 제한적
- jhgan/ko-sroberta-multitask는 2022년 모델로 현재는 레거시. 단문 유사도엔 무방하나 RAG엔 교체 권장
